# Minekea Demo World — Tiled Layout Plan (126×126)

Every Minekea block, placed into an **8×8 grid of 16×16 tiles** inside a pre-built
**126×126** arena. Each tile is a **14×14 polished-andesite pad** framed by
**smooth-stone** borders; the touching borders of neighbouring tiles form the
2-block aisles. One family per tile (big families get a merged 2-tile room),
and a teleport command block on every tile for clean per-tile screenshots.

- **Blocks placed:** 2,584 (2,477 distinct blocks + 107 glass-jar displays)
- **Rooms:** 34 · **Tiles used:** 39 / 64 · **Free for growth:** 25 tiles (rows 5–7 are empty)
- **Arena:** front-left `(-191, 55, 190)` → back-right `(-66, 55, 65)`, floor y=55
- **Floor:** `polished_andesite` (tile interiors) · **Borders:** `smooth_stone`
- Everything below is generated by [`generate_layout.py`](./generate_layout.py) —
  rerun it whenever blocks are added and the manifest, tile map, stats, and build
  function all update together.

## 1. Coordinate system & tile geometry

```
        X = -191 (west/left) ───────────────────────────► X = -66 (east/right)
  Z=190  ┌───────────────────────────────────────────────┐  FRONT  (you enter here)
  front  │ r0  BUILDING …                                 │
         │ r1  …                                          │  each cell = one 16×16
         │ r2  … FURNITURE …                              │  tile (14×14 usable)
         │ r3  … STORAGE / CONTAINERS …                   │
         │ r4  … GLASS JARS / DECORATIONS …               │  -Z = back = north
  Z=65   └───────────────────────────────────────────────┘  BACK
```

- **X**: −191 (west/**left**) → −66 (east/**right**), 126 wide. `+X` = east.
- **Z**: 190 (**front**, near) → 65 (**back**, north/far), 126 deep. `−Z` = back.
- **Y**: floor blocks at **y=55**; every display block sits at **y=56**; compressed
  stacks rise **y=56…64**.

**Tile `(col c = 0…7 west→east, row r = 0…7 front→back)`** interior:

| axis | range | width |
|------|-------|------:|
| X | `-191 + 16c` … `-178 + 16c` | 14 (`lx` 0…13) |
| Z | `190 - 16r` … `177 - 16r` | 14 (`lz` 0…13) |

The **front row of each tile (`lz = 0`, i.e. Z = `190 − 16r`) is left clear** — it's
the standing row that holds the label sign (front-left corner) and the teleport
command block (front-right corner). Displays fill `lz = 1…13` (14 wide × 13 deep =
**182 cells** per single tile; a merged 2-tile room is 30 × 13 = **390**).

Block-placement formula (what the generator emits):

```
X = -191 + 16*col + lx          (lx = 0..width-1, centred in the room)
Z = 190  - 16*row - lz          (lz = 1..13,  front row lz=0 reserved)
Y = 56   + y_offset             (y_offset 0 for normal blocks, 0..8 for compressed)
```

### Command blocks & teleport (all 64 tiles)

A command block sits at the **front-right corner** of every tile:

```
position  = (-178 + 16*col, 55, 190 - 16*row)     # your -178,55,190 + every 16
block     = command_block[facing=up]{Command:"tp @p ~-6.5 ~1 ~ 180 0", auto:0b}
trigger   = stone_pressure_plate on top (y=56)
```

The command is **identical on every tile** because its coordinates are relative to
the command block: `~-6.5` west of the front-right corner lands you at **X = tile
centre (−184.5 + 16c)**, `~1` up puts you on the floor (**y=56**), same Z (the front
row), yaw **180 = facing north into the tile**, pitch 0. Step on the plate → you're
standing front-and-centre looking at that tile's display. Empty tiles get command
blocks too (they just look at an empty pad), so screenshot spots already exist for
future families.

## 4. Compressed blocks — spread by material type

`building/compressed` is 146 materials × 9 tiers. Rather than one dense tile, the
146 materials are split across **6 tiles by type**, each a walkable grove of 9-tall
stacks (tier `1x` on the floor → `9x` on top):

| Tile | Group | Count | Examples |
|------|-------|------:|----------|
| (4,1) | Stone | 68 | stone, andesite, bricks, deepslate, sandstone, basalt, obsidian… |
| (5,1) | Wood | 22 | 11 plank types + 11 log/stem types |
| (6,1) | Ore/Metal | 8 | iron, gold, diamond, netherite, copper, lapis, redstone, amethyst |
| (7,1) | Terracotta | 16 | the 16 dye colours |
| (0,2) | Concrete | 16 | the 16 dye colours |
| (1,2) | Glazed Terracotta | 16 | the 16 dye colours |

Column-style materials (logs, basalt, quartz…) carry an `axis` state; their default
placement is vertical, which is what the build function uses.

## 5. Glass jars — items, fluids & mobs

The glass jar is a **single registered block** (`minekea:containers/glass_jar`)
whose display depends on NBT, so it gets two tiles:

- **(2,4) Items — 95 jars:** every id in `GlassJarBlock.ALLOWED_ITEM_IDS`
  (46 vanilla items + 16 dyes + 16 concrete powders + 17 Minekea wax).
- **(3,4) Fluids & Mobs — 12 jars:** 1 empty + 4 fluids (water, lava, milk, honey)
  + 7 mobs (allay, bat, bee, endermite, silverfish, **tiny** slime, vex).

The contents ride on two manifest columns, **`contents`** (the id stored) and
**`contents_label`** (a display name):

```
block_id,x,y,z,tile_col,tile_row,room,zone,tier,state,label,contents,contents_label
minekea:containers/glass_jar,-159,56,125,2,4,jars_items,CONTAINERS,,facing=south,Glass Jars - Items (95),minecraft:amethyst_shard,Amethyst Shard
minekea:containers/glass_jar,-142,56,125,3,4,jars_misc,CONTAINERS,,facing=south,Glass Jars - Fluids & Mobs (12),,Empty Jar
```

**The build fills the jars** via block-entity NBT matching `GlassJarBlockEntity`'s
persisted keys — so all 106 non-empty jars show their contents on load:

| kind | NBT written by `demo_build.mcfunction` |
|------|----------------------------------------|
| item  | `{StoredItem:"<id>",StoredItemQty:64,FullItemStacks:7}` (full jar) |
| fluid | `{StoredFluid:"<id>",StoredFluidAmount:8.0d}` (full = `MAX_BUCKETS`) |
| mob   | `{"minecraft:entity_data":{id:"<entity>",<real fields>}}` — see below |

The list is regenerated from the mod source by
[`extract_jar_contents.py`](./extract_jar_contents.py); the NBT builder lives in
`generate_layout.py` (`jar_nbt`, mob fields in `MOB_NBT`).

> **Mobs need real entity NBT, not just an id.** `GlassJarBlockEntity` wraps the stored
> mob in a `TypedEntityData` whose constructor strips `id` from the tag, and `hasMob()`
> only reports the mob present when that id-less tag is **non-empty**. A real in-game
> capture always carries `Health`/`Brain`/`Motion`/… so it never is — but the earlier
> bare `{id:"<entity>"}` stripped down to `{}`, so every mob jar rendered empty. Each mob
> now ships at least one real field (`Health`, `Size:0` for the tiny slime, etc.); the
> renderer fakes `Pos`, normalises facing and drops equipment, so `UUID`/`Pos`/`Motion`
> are intentionally omitted. This is a demo-data requirement, **not** a renderer bug.

> **Renderer prerequisite:** building this demo surfaced a client-crash bug — the jar
> renderer only handled honey/milk and threw on any other fluid (so water and lava
> jars hard-crashed on render). Fixed in `{Fabric,NeoForge}GlassJarBlockEntityRenderer`
> to render water, lava, milk, honey explicitly and fall back to water for anything
> else (never throw). This demo assumes that fix is present.

## 6. Building it — `demo_build.mcfunction`

One runnable function lays the whole gallery inside the existing arena:

1. **Floor base** — one `fill` of smooth stone across the 128×128 (arena + 1-block
   outer border ring).
2. **Tile interiors** — `fill` polished andesite for each of the 39 used tiles
   (plus the seams of merged rooms).
3. **Display blocks** — a `setblock` for all 2,584 blocks (absolute coords),
   oriented to face the viewer (§6a), plus 24 upper halves for doors/armoires and
   NBT contents for the 106 filled jars — **2,608 setblocks** total.
4. **Label signs** — an `oak_sign` (facing south, toward the viewer) at each room's
   front-left corner with the family name + count.
5. **Command blocks + plates** — all 64 teleport pads (§1).

Install as a datapack function and run once:

```
<world>/datapacks/minekea_demo/
  pack.mcmeta            {"pack":{"pack_format":<match your MC version>,"description":"Minekea demo"}}
  data/minekea/function/build.mcfunction     ← copy demo_build.mcfunction here
```
(`data/minekea/functions/` with an `s` on MC versions before 1.21.)
```
/reload
/function minekea:build
```

Or paste the commands directly (they're plain `/fill` and `/setblock`). The arena,
its outer smooth-stone border, and floor level are assumed to exist already at the
coordinates above; re-running is idempotent.

### 6a. Block facing

The viewer stands at each tile's front (high Z) looking **north (−Z)**, so directional
blocks are placed **facing south** (front toward the camera). Because families are
heterogeneous, facing is applied **per block** (`generate_layout.py` → `block_state`),
recorded in the manifest's **`state`** column:

| Blocks | State applied |
|--------|---------------|
| stairs (normal + bookshelf) | `facing=north` (rotated 180 from the viewer) |
| **vertical** slabs only | `facing=east` (90 CCW from south) |
| covers · chairs · barrels · glass jars | `facing=south` |
| armoires (2-tall) | `facing=north,half=lower` **+ an upper half above** (rotated 180) |
| doors (2-tall) | `facing=south,half=lower` **+ an upper half placed above** |
| trapdoors | `facing=south,half=bottom,open=true` (stood up to be seen) |
| horizontal slabs, stools, pillows, tables, shelves, shutters, display cases | left default (already face the viewer or are symmetric) |

Horizontal slabs use a `type` property (not `facing`), which is exactly why the rule
is per-block — a blanket `facing=south` on the `slabs` family would emit invalid
states and those `setblock`s would silently fail.

## 7. Growth — "plenty of room"

- **25 empty tiles** (rows 5–7, plus (7,4)) are reserved and already have working
  teleport command blocks. Drop a new family onto any empty tile — no reflow.
- A **new wood type** adds ~1 cell to each affected family; every family tile has
  spare rows (a single tile holds 182 cells; the biggest current family, dyed, uses
  192 across its 2-tile room). Nothing needs to move.
- A **new compressed material** adds one stack to its type tile (each type tile is
  far from full).
- **Regenerate:** refresh `block_inventory.txt` (§8), run
  `python extract_jar_contents.py` if jar contents changed, then
  `python generate_layout.py`. The tile map, manifest, stats, and build function all
  update together.

### Refreshing the inventory after adding blocks
```bash
# after a build/datagen has run:
cd minekea/common/src/main/generated/assets/minekea/blockstates
find . -name '*.json' | sed 's|^\./||;s|\.json$||' | sort \
  > /path/to/minekea/demo-world/block_inventory.txt
# the hand-authored specials are re-added automatically by the generator
```
