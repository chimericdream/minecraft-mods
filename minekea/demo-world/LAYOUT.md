# Minekea Demo World — Block Layout Plan

A placement plan for showing **every registered Minekea block** in a single
walkable gallery, sized to a fixed **126-wide** plot whose depth grows one chunk
(16 blocks) at a time: `126×30 → 126×46 → 126×62 → 126×78 → …`.

- **Total blocks placed:** 2,584 (2,477 distinct blocks + 107 glass-jar displays)
- **Footprint cells used:** 1,416 (compressed blocks are stacked, see below)
- **Depth used:** 77 rows → **recommended build size: `126 × 78`**
- Every number, coordinate, and the map below is generated by
  [`generate_layout.py`](./generate_layout.py) — rerun it whenever blocks are
  added and everything updates together.

> **Design decisions (locked with the mod author):** organize by
> *category → family zones*; **one instance per block**; **walkable gallery with
> per-family sign labels**; deliver a **Markdown plan + CSV coordinate manifest**.

---

## 1. Coordinate system

```
        X = 0 ────────────────────────────────► X = 125   (126 wide, fixed)
   Z=0   ┌───────────────────────────────────────────────┐   NORTH (entrance)
         │  BUILDING     zone                             │
         │  FURNITURE    zone                             │
         │  STORAGE      zone                             │   depth grows south,
         │  CONTAINERS   zone                             │   +16 at a time
   Z=77  └───────────────────────────────────────────────┘   SOUTH
```

- **X** runs west→east, `0…125`, **fixed at 126**.
- **Z** runs north→south and is the **expandable** axis (`30/46/62/78/…`).
- **Y** is height. The walkable floor is at **Y = −1** (a slab/path course);
  blocks sit at **Y = 0**. The only blocks that use height are **compressed**
  blocks, which stack **Y = 0…8** (tier `1x` at the bottom → `9x` on top).
- A **1-block perimeter walkway** rings the whole plot (`.` in the map).

Pick the entrance at the north edge (Z=0) looking south; the zones read like
chapters as you walk in.

## 2. Zone summary

Zones are laid out north→south in the mod's own taxonomy / creative-tab order.
Each zone opens with a 2-row header band (sign wall + walking lane) and closes
with a **growth-reserve lane** (empty rows) so new families can be added without
reflowing the zones below it.

| # | Zone | Z range | Footprint cells | Blocks | Growth reserve |
|---|------|--------:|----------------:|-------:|---------------:|
| 1 | **BUILDING**    | 1 – 27  | 950   | 2,118 | 6 rows |
| 2 | **FURNITURE**   | 27 – 43 | 254   | 254   | 5 rows |
| 3 | **STORAGE**     | 43 – 52 | 45    | 45    | 3 rows |
| 4 | **CONTAINERS**  | 52 – 60 | 37    | 37    | 3 rows |
| 5 | **GLASS JARS**  | 60 – 69 | 107   | 107   | 3 rows |
| 6 | **DECORATIONS** | 69 – 76 | 23    | 23    | 2 rows |

BUILDING holds 950 footprint cells but **2,118 blocks** because the compressed
family is stacked (see §4). GLASS JARS is 107 copies of *one* block distinguished
by contents (see §5).

## 4. Compressed blocks — the special case

`building/compressed` is **146 materials × 9 tiers = 1,314 blocks**, 53% of the
whole mod. Laid flat that would eat a third of the floor. Instead each material
is a **1×1 footprint, 9-tall vertical stack**, tier `1x` on the floor up to `9x`
at the top. This:

- keeps the demo compact (146 cells instead of 1,314),
- reads instantly — a single glance up a column shows all compression tiers,
- still places **one instance of every block** (nothing is omitted).

Column-style materials (logs, basalt, quartz, etc.) use the same stacks; they
carry an `axis` state, so place them oriented vertically.

The `storage/compressed` family (apple, beetroot, charcoal, nugget sacks, …) are
**single** blocks, not tiered — they stay flat in the STORAGE zone.

## 5. Glass jars — their own section

The glass jar is a **single registered block** (`minekea:containers/glass_jar`)
whose displayed contents live in NBT, so one cell would show almost nothing.
It gets a dedicated zone with **one jar per thing it can hold — 107 jars**:

| Group | Count | What it is |
|-------|------:|------------|
| Empty jar | 1 | the plain block, nothing stored |
| **Items** | 95 | everything in `GlassJarBlock.ALLOWED_ITEM_IDS` |
| **Fluids** | 4 | water, lava, milk, honey |
| **Mobs** | 7 | allay, bat, bee, endermite, silverfish, slime *(tiny only)*, vex |

The item list is the 46 vanilla items (apple, coal, redstone, sand, …) plus the
16 dyes, 16 concrete powders, and 17 Minekea wax ingredients. Mobs come from
`GlassJarItem.canCaptureMob()`; fluids from the jar's bucket/bottle handling.

Because all 107 rows share the same `block_id`, the manifest carries two extra
columns — **`contents`** (the item/fluid/mob id to store) and
**`contents_label`** (a display name for the sign):

```
block_id,x,y,z,zone,family,tier,label,contents,contents_label
minekea:containers/glass_jar,1,0,63,GLASS JARS,jars_empty,,Empty Jar,,Empty Jar
minekea:containers/glass_jar,4,0,63,GLASS JARS,jars_items,,Jars - Items,minecraft:amethyst_shard,Amethyst Shard
minekea:containers/glass_jar,54,0,63,GLASS JARS,jars_fluids,,Jars - Fluids,minecraft:water,Water
minekea:containers/glass_jar,60,0,63,GLASS JARS,jars_mobs,,Jars - Mobs,minecraft:allay,Allay
```

Building notes:
- **Items/fluids** — fill via the jar's own interaction (right-click with the
  item / a filled bucket or bottle), or set the block entity's stored stack /
  `StoredFluid` + `StoredFluidAmount` directly.
- **Mobs** — captured by right-clicking the mob with the jar *item*; the mob
  lives on the stack's vanilla `ENTITY_DATA` component. Only a **tiny** slime
  qualifies. Jarred allay/bat/bee/endermite/silverfish/slime/vex also play a
  faint ambient sound, so keep the mob row on the aisle where visitors pass.
- The two mod fluids (milk, honey) also appear as **cauldrons** one zone north
  (CONTAINERS, z 55) — same fluids, shown in both container forms.

`glass_jar_contents.csv` is the data behind this section; regenerate it with
`extract_jar_contents.py`, which reads the lists straight out of the Java source
so it can't drift from the mod.

## 7. How growth works ("leave room for new families")

The gallery is built to absorb new content without a redesign:

1. **New wood type** (e.g. a future "willow"): adds ~1 cell to each of ~11
   families (slabs, stairs, beams, covers, vertical slabs, armoires, bookshelves,
   display_cases, seating×2, shelves×2, shutters×2, tables, barrels, crates,
   doors, trapdoors). Each family's grid gains a cell on its last row (or a new
   row); the **growth-reserve lanes** at the bottom of BUILDING (6 rows) and
   FURNITURE (5 rows) absorb the extra depth. One full new wood ≈ +1–2 rows per
   affected zone.
2. **New compressed material**: adds one vertical stack in the compressed grid —
   1 footprint cell, no new families.
3. **A brand-new family**: append it to the correct zone; if the zone's shelf is
   full it wraps into the reserve lane. If a zone overflows its reserve, bump the
   plot to the next depth (`126×78 → 126×94`) — only the zones **south** of the
   change shift; the reserve lanes minimize even that.
4. **Regenerate**: edit `block_inventory.txt` (or re-export it, §7) and run
   `python generate_layout.py`. The CSV, stats, and map all update in lockstep,
   and the script reprints the new recommended plot size.

Ordering rationale: fastest-growing zones (BUILDING, FURNITURE) are north with
the largest reserves; the small, stable zones sit south where a shift costs the
least.

### Regenerating the inventory after adding blocks
`block_inventory.txt` is the list of generated blockstate paths plus the
hand-authored blocks. To refresh it from the mod after adding/removing blocks:

```bash
# from repo root, after a build/datagen has run:
cd minekea/common/src/main/generated/assets/minekea/blockstates
find . -name '*.json' | sed 's|^\./||;s|\.json$||' | sort \
  > /path/to/minekea/demo-world/block_inventory.txt
# (the 7 hand-authored specials are re-added automatically by the generator)
```

Then `python generate_layout.py` in this folder. (If jar contents changed, run
`python extract_jar_contents.py` first.)

## 9. CSV manifest quick reference

```
block_id,x,y,z,zone,family,tier,label,contents,contents_label
minekea:building/beams/acacia,1,0,4,BUILDING,beams,,Beams,,
minekea:building/compressed/oak_planks/1x,103,0,16,BUILDING,compressed,1x,,,
minekea:building/compressed/oak_planks/9x,103,8,16,BUILDING,compressed,9x,,,
minekea:containers/glass_jar,4,0,63,GLASS JARS,jars_items,,Jars - Items,minecraft:amethyst_shard,Amethyst Shard
```

- `x, y, z` are plot-local (add your world origin offset when building).
- `y = 0` is ground level (floor slab at `y = −1`); compressed tiers use `y = 0…8`.
- `tier` is set only for compressed stacks (`1x`…`9x`).
- `label` is non-empty on the first block of each family — put a sign there.
- `contents` / `contents_label` are set only for glass jars (§5) — the id to
  store and a display name; empty for every other block.
- Directional blocks (doors, shutters, trapdoors, stairs, tables, seating) are
  placed one-per-block in a neutral facing; face them toward the north aisle so
  the front reads to a visitor walking the lane.
