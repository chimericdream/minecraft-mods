# Minekea Demo World — By-Material, Tiered Layout

Every Minekea block, **grouped by material** and laid on a **rising staircase** so nothing
is hidden directly behind anything else. The one rule that drives the whole build:

> every display block sits at **`y = 56 + depth`** (depth = how many rows back it is),
> on a solid support column — so each row back is one block higher.

- **Regions:** 54 · **Display blocks:** 2,560 (+24 door/armoire upper halves) · **Jars filled:** 106
- **Arena:** front-left corner **`(-191, 55, 191)`** · `X[-191..-66]` (126 wide) · `Z[99..191]` (93 deep)
- **Floor:** `polished_andesite` region pads + full-width tier platforms over a `smooth_stone` base
- Generated by [`generate_layout.py`](./generate_layout.py) — rerun after adding blocks and
  the manifest, region list, stats, and build function all update together.

---

## 1. Coordinate system

```
        X = -191 (west/left) ─────────────────────────────► X = -66 (east/right)
  Z=191  ┌─────────────────────────────────────────────────┐  FRONT (corner -191,55,191)
  front  │ row0:  Oak Spruce Birch … Pale Oak               │
         │ row1:  Bamboo Crimson Warped  Dyed  Colored …    │  regions flow left→right,
         │ row2:  … stones …                               │  wrapping north with a 3-block
         │ …      … metals … food … jars                   │  aisle between rows
  Z=99   └─────────────────────────────────────────────────┘  BACK (north)
```

- **X**: −191 (west/left) → −66 (east/right). `+X` = east.
- **Z**: 191 (front/south) → 99 (back/north). `−Z` = back. Rows wrap north.
- **Y**: floor **y=55**; a block at depth *d* sits at **y=56+d**, supported from below.

Regions flow west→east until a row passes `ROW_WIDTH` (124), then drop to a new row
`max-depth + 3` blocks north (the **3-block row aisle**). 2-block gaps separate regions
within a row. Full geometry is in [`layout_regions.txt`](./layout_regions.txt); every
block's absolute coords in [`demo_layout_manifest.csv`](./demo_layout_manifest.csv).

### Per-region label + teleport
Each region's front-left has an `oak_sign` (material name + count) and a
`command_block` + `stone_pressure_plate` that teleports you to a vantage **2 blocks in
front** of the region, centred, facing north (`tp @p <centre-x> 56 <front+2> 180 0`). The
vantage is clamped to the arena floor and lands in the row aisle, so it never drops you
into the void or inside the next row's blocks.

## 2. Everything is tiered

Blocks are **not** laid flat. Each row back steps up one block (with a `polished_andesite`
support column beneath), so from the front you see every block — none is hidden behind
another. This applies to the whole set, not just compressed:

- **Flat blocks** (stairs, slabs, furniture…) fill staircase rows (12 wide) rising back.
- **Compressed blocks** form small **stepped 3×3 podiums** behind the flat rows — `1-3x`
  on their front step, `4-6x` one back+up, `7-9x` again — and each extra podium-row steps
  up 3 more so back podiums clear front ones.

The **colour regions are the exception — they stay flat** (all at y=56); their gradients
recede back without rising (see §4).

### Consistent block-family positions (wood & stone-like sets)
Within the wood sets — and within the stone-like sets (those with stairs/slabs/bookshelves)
— each block family sits on a **fixed band/tier**, so the armoire, stairs, table, bookshelf,
… are always in the **same relative position**, easy to find and compare across materials.
Bands run front→back (`generate_layout.py` → `WOOD_BANDS` / `STONE_BANDS`), e.g. woods:
surfaces → slabs → stairs → shelving → seating/tables → storage/deco (shutters, warped wart, …)
→ **armoires & doors**. The 2-tall families get their **own dedicated back band**, and the
staircase uses **cumulative elevation** — each tier rises by the height of the tier in front
of it (2 blocks over a double-tall tier, else 1) — so a double-height block never hides what's
behind it (the podiums behind the armoire band step up to its top). Woods reserve a fixed
width per family for exact column alignment (a wood missing a family, like oak's bookshelves,
just leaves that spot empty); the stone-like sets align by tier + order (their brick-variant
counts differ too much to column-align without big gaps).

## 3. Material grouping

`generate_layout.py` → `canonical(path)` buckets each **non-colour** block into a material
region (special families → woods → stones/minerals/metals via ordered substring rules,
metals before the generic `stone` so `redstone_block` isn't swallowed). A material's own
compressed tiers fold into its region as podiums; only compressed-only materials that
belong to nothing (metals/gems, and soil/sand → **Soil & Sand**) form their own regions.
An **unmapped material raises** — nothing silently vanishes. `is_colored(path)` peels off
colour blocks first (§4).

## 4. Colour regions — by block type, full gradient

Colour blocks are organised **by block type**, each type a column showing the **full
16-colour gradient** receding back. Unlike the material regions, **colour regions stay
flat** (all at y=56 — no tiering). Colour order is Minecraft's current creative order:

> white · light_gray · gray · black · brown · red · orange · yellow · lime · green ·
> cyan · light_blue · blue · purple · magenta · pink

A block type with an **un-dyed variant** (wax, votive candles) puts that **plain** one in
**spot 0**, ahead of white. Votive candles are placed with **4 candles per block**
(`candles=4`). The colour regions:

| Region | Columns (block types) | Blocks |
|--------|-----------------------|-------:|
| Dyed Blocks | the 12 dyed bases (bone_block, stone, prismarine, …) | 192 |
| Colored Building | beams/covers × concrete/terracotta/glazed | 96 |
| Colour Accents | pillows, dyes, votive candles (+plain), wax (+plain) | 66 |
| Compressed Concrete | 9 tier-columns (1x…9x), each a colour gradient | 144 |
| Compressed Terracotta | 9 tier-columns | 144 |
| Compressed Glazed Terracotta | 9 tier-columns | 144 |

## 5. Glass jars — items, fluids & mobs

The glass jar is a single registered block whose display depends on NBT, so it keeps its
own two regions (built from [`glass_jar_contents.csv`](./glass_jar_contents.csv)):
**Items (95)** and **Fluids & Mobs (12)** — 1 empty + 4 fluids (water, lava, milk, honey)
+ 7 mobs. The build fills each via block-entity NBT (`generate_layout.py` → `jar_nbt`):

| kind | NBT written |
|------|-------------|
| item  | `{StoredItem:"<id>",StoredItemQty:64,FullItemStacks:7}` (full jar) |
| fluid | `{StoredFluid:"<id>",StoredFluidAmount:8.0d}` |
| mob   | `{"minecraft:entity_data":{id:"<entity>",<real fields>}}` |

> **Mobs need real entity NBT, not just an id.** `TypedEntityData`'s ctor strips `id`, and
> `hasMob()` needs the id-less tag non-empty, so a bare `{id}` rendered empty. Each mob ships
> ≥1 real field (`Health`, `Size:0` for the tiny slime, …). Demo-data requirement, not a bug.

> **Renderer prerequisite:** the jar renderer was fixed to render water/lava/milk/honey and
> fall back to water for anything else (never throw). This demo assumes that fix is present.

## 6. Block facing

Viewer stands at each region's front looking **north (−Z)**; facing is per-block
(`generate_layout.py` → `block_state`), in the manifest's `state` column:

| Blocks | State |
|--------|-------|
| votive candles | `candles=4,lit=false` (4 candles per block) |
| **vertical** stairs (`.../vertical/...`) | `facing=south` (face the viewer) |
| other stairs (normal + bookshelf) | `facing=north` |
| **vertical** slabs only | `facing=east` |
| covers · chairs · barrels · glass jars | `facing=south` |
| armoires (2-tall) | `facing=north,half=lower` **+ upper half above** |
| doors (2-tall) | `facing=south,half=lower` **+ upper half above** |
| trapdoors | `facing=south,half=bottom,open=true` |
| horizontal slabs, stools, pillows, tables, shelves, shutters, display cases | default |

**Shutters** show only the real (closed) block; `*_open` `OpenShutterHalfBlock`s are skipped
(internal open-halves; standalone they render half-width and used to crash on click — also
guarded in the mod now, covered by `OpenShutterHalfBlockGameTest`).

## 7. Building it — `demo_build.mcfunction`

0. **Clear** — eight `fill … air` commands wipe the 128×128 footprint (from the front-left
   corner) up ~15 blocks, then `kill @e[type=item]` removes anything knocked loose, so a
   re-run starts clean. (Each fill is 128×128×2 = 32768 blocks, the `/fill` maximum.)
1. **Floor base** — one `fill` of smooth stone over the arena + border.
2. **Region pads** — a `fill` of polished andesite per region footprint.
3. **Tier platforms** — a full-width `polished_andesite` step per tier (`fill`ed), so every
   region reads as one solid stairstep; display blocks sit on top and empty columns just show
   the step surface (no filler display blocks). Steps jump up 2 over a double-tall tier.
4. **Display blocks** — all 2,560 setblocks (facing state + jar NBT + 24 upper halves).
5. **Signs + command blocks + plates** — one per region.

Install as a datapack function (`data/minekea/function/build.mcfunction`), `/reload`, then
`/function minekea:build`. Idempotent (`/fill` + `/setblock`).

### Refreshing after adding blocks
```bash
cd minekea/common/src/main/generated/assets/minekea/blockstates
find . -name '*.json' | sed 's|^\./||;s|\.json$||' | sort > /path/to/minekea/demo-world/block_inventory.txt
python generate_layout.py     # (run extract_jar_contents.py first if jar contents changed)
```
A new material needs a `canonical()` rule (+ `ORDER_*` slot), or `is_colored()` handling; an
unmapped material raises rather than disappearing.
