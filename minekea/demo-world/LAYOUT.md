# Minekea Demo World — By-Material Layout

Every Minekea block, **grouped by material** into flexible-size regions: all the oak
things together, all the granite things together, all the black things together, and
so on. Each region sizes itself to its contents, and regions flow left→right in a
walkable order. Compressed blocks are shown as small **stepped 3×3 podiums** rather
than tall towers.

- **Regions:** 64 · **Display blocks:** 2,560 (+24 door/armoire upper halves) · **Jars filled:** 106
- **Arena:** `X[-192..-67]` (126 wide) · `Z[100..195]` (96 deep) · floor **y=55**, displays **y=56**
- **Floor:** `polished_andesite` region pads over a `smooth_stone` base (aisles + front apron)
- Everything is generated by [`generate_layout.py`](./generate_layout.py) — rerun it
  whenever blocks are added and the manifest, region list, stats, and build function
  all update together.

> Replaces the earlier **tiled** model (8×8 grid grouped by block *family*). The old
> generator is in git history if you ever want to diff it.

## 1. Coordinate system

```
        X = -192 (west/left) ─────────────────────────────► X = -67 (east/right)
  Z=195  ┌─────────────────────────────────────────────────┐  FRONT (viewing apron)
  front  │ row0:  Oak  Spruce  Birch … Pale Oak             │
         │ row1:  Bamboo Crimson Warped  White Orange …     │  regions flow left→right,
         │ row2:  … Pink Gray … (colours) …                 │  wrapping to a new row
         │ …      … stones … metals … earth/food …          │  further north each time
  Z=100  └─────────────────────────────────────────────────┘  BACK (north)
```

- **X**: −192 (west/**left**) → −67 (east/**right**). `+X` = east. Regions flow east.
- **Z**: 195 (**front**, near/south) → 100 (**back**, far/north). `−Z` = back. Rows wrap north.
- **Y**: floor **y=55**; flat displays **y=56**; podium tiers rise **y=56 / 57 / 58**.

Regions are packed by a simple shelf flow: place left→right until a row passes
`ROW_WIDTH` (124), then drop to a new row `max-region-depth + 2` blocks north. A
2-block aisle separates regions. The authoritative placement of every region (front-left
corner + footprint) is in [`layout_regions.txt`](./layout_regions.txt); every block's
absolute coords are in [`demo_layout_manifest.csv`](./demo_layout_manifest.csv).

### Per-region label + teleport
Each region has, at its front-left:
- an `oak_sign[rotation=0]` (faces south, toward you) with the material name + block count;
- a `command_block[facing=up]` + `stone_pressure_plate` that teleports you to a vantage
  a few blocks in front of the region, centred and facing north:
  `tp @p <region-centre-x> 56 <region-front+4> 180 0` (yaw 180 = facing north).

Unlike the old tiled model these are **absolute** per-region teleports (regions vary in
width), and the front row carries a floored **viewing apron** so front-row vantages never
drop you into the void.

## 2. Material grouping (how blocks are bucketed)

`generate_layout.py` → `canonical(path)` maps each block to a material region. The
order of rules matters (first match wins):

1. **Special families:** `storage/compressed/*` → **Compressed Food**; `decorations/lighting`
   → **Lighting**; `crops/warped_wart` → **Warped**; cauldrons/egg-crates/`fake_cake`/plain
   wax → **Misc**; `glass_jar` → the dedicated jar regions.
2. **Colours** (dyed blocks, pillows, dye storage, votive candles, and
   `<colour>_concrete/terracotta/glazed_terracotta` compressed) → the 16 colour regions,
   in the order **white, orange, magenta, light_blue, yellow, lime, pink, gray, light_gray,
   cyan, purple, blue, brown, green, red, black**.
3. **Woods** (a wood + its `stripped_`, `_log`, `_planks`, `_stem`, `_mosaic`, framed planks,
   and compressed planks/logs) → the 12 wood regions. Careful cases handled: `dark_oak`/
   `pale_oak` match before `oak`; `crimson_basalt_bricks`/`warped_nether_brick` are **not**
   treated as wood (they fall through to the stone rules).
4. **Stones / minerals / metals** via ordered substring rules, e.g. all `*basalt*` → basalt,
   all `*deepslate*` → deepslate, `iron_block` → iron, etc. Metals are matched **before** the
   generic `stone` rule (so `redstone_block` doesn't get swallowed by "stone").
5. Anything left raises an error, so a new material can never silently vanish.

This "fold-in" means a stone/colour/wood material carries its **own compressed tiers**
inside its region; only compressed-only materials that belong to nothing (metals/gems,
plus soil/sand-type blocks in **Soil & Sand**) form their own small regions.

## 3. Compressed blocks — stepped 3×3 podiums

Each compressed material is 9 tiers (`1x`…`9x`). Instead of a 9-tall tower (ugly) or a
flat plane (also ugly), each material becomes a **3×3 stepped podium**, 3 blocks tall:

```
  tier +2 (y=58):        [7x][8x][9x]     ← furthest back (north)
  tier +1 (y=57):     [4x][5x][6x]
  tier  0 (y=56):  [1x][2x][3x]           ← front, on the floor
```

Each higher tier steps one block **back** (−Z) and one **up**, with a solid
`smooth_stone` support column under every raised block so it reads as a staircase. A
region lays its podiums in rows of up to 3, behind its flat blocks. (1,314 support
blocks total = 146 podiums × 9.)

## 4. Glass jars — items, fluids & mobs

The glass jar is a **single registered block** (`minekea:containers/glass_jar`) whose
display depends on NBT, so it keeps its own two regions (built from
[`glass_jar_contents.csv`](./glass_jar_contents.csv), not the block inventory):

- **Glass Jars – Items — 95 jars:** every id in `GlassJarBlock.ALLOWED_ITEM_IDS`.
- **Glass Jars – Fluids & Mobs — 12 jars:** 1 empty + 4 fluids (water, lava, milk, honey)
  + 7 mobs (allay, bat, bee, endermite, silverfish, **tiny** slime, vex).

The build fills the jars via block-entity NBT (`generate_layout.py` → `jar_nbt`):

| kind | NBT written |
|------|-------------|
| item  | `{StoredItem:"<id>",StoredItemQty:64,FullItemStacks:7}` (full jar) |
| fluid | `{StoredFluid:"<id>",StoredFluidAmount:8.0d}` (full = `MAX_BUCKETS`) |
| mob   | `{"minecraft:entity_data":{id:"<entity>",<real fields>}}` — see below |

> **Mobs need real entity NBT, not just an id.** `GlassJarBlockEntity` wraps the mob in a
> `TypedEntityData` whose constructor strips `id` from the tag, and `hasMob()` only reports
> the mob present when that id-less tag is **non-empty**. A real capture always carries
> `Health`/`Brain`/`Motion`/… so it never is — a bare `{id:"<entity>"}` stripped to `{}` and
> the jar rendered empty. Each mob now ships ≥1 real field (`Health`, `Size:0` for the tiny
> slime, etc.); the renderer fakes `Pos`, normalises facing and drops equipment, so
> `UUID`/`Pos`/`Motion` are omitted. Demo-data requirement, **not** a renderer bug.

> **Renderer prerequisite:** building this demo surfaced a client-crash bug — the jar
> renderer only handled honey/milk and threw on any other fluid (water/lava jars hard-crashed
> on render). Fixed in `{Fabric,NeoForge}GlassJarBlockEntityRenderer` to render water, lava,
> milk, honey explicitly and fall back to water otherwise (never throw). This demo assumes
> that fix is present.

## 5. Block facing

The viewer stands at each region's front (high Z) looking **north (−Z)**, so directional
blocks are oriented per block (`generate_layout.py` → `block_state`), recorded in the
manifest's **`state`** column:

| Blocks | State applied |
|--------|---------------|
| stairs (normal + bookshelf) | `facing=north` (rotated 180 from the viewer) |
| **vertical** slabs only | `facing=east` (90° CCW from south) |
| covers · chairs · barrels · glass jars | `facing=south` |
| armoires (2-tall) | `facing=north,half=lower` **+ an upper half above** |
| doors (2-tall) | `facing=south,half=lower` **+ an upper half above** |
| trapdoors | `facing=south,half=bottom,open=true` (stood up to be seen) |
| horizontal slabs, stools, pillows, tables, shelves, shutters, display cases | left default |

Horizontal slabs use a `type` property (not `facing`), which is why the rule is
per-block — a blanket `facing=south` on the `slabs` family would emit invalid states.

**Shutters** show only the real (closed) block. The `*_open` `OpenShutterHalfBlock`s are
skipped: they're internal open-state halves that only exist flanking an opened shutter, so
standalone they render half-width and **used to crash on click**. That crash is also guarded
in the mod now (`OpenShutterHalfBlock`'s `useWithoutItem`/`playerWillDestroy` bail out when
the centre block isn't a shutter), covered by
`minekea/fabric/src/gametest/.../OpenShutterHalfBlockGameTest`.

## 6. Building it — `demo_build.mcfunction`

One runnable function lays the whole gallery:

1. **Floor base** — one `fill` of smooth stone over the arena (+ 1-block border + front apron).
2. **Region pads** — a `fill` of polished andesite per region footprint.
3. **Podium supports** — the 1,314 smooth-stone step columns under raised compressed tiers.
4. **Display blocks** — a `setblock` for all 2,560 blocks (facing state + jar NBT + 24 upper halves).
5. **Label signs** — one `oak_sign` per region.
6. **Command blocks + plates** — one teleport pad per region.

Install as a datapack function and run once:

```
<world>/datapacks/minekea_demo/
  pack.mcmeta                                {"pack":{"pack_format":<your MC version>,"description":"Minekea demo"}}
  data/minekea/function/build.mcfunction     ← copy demo_build.mcfunction here
/reload
/function minekea:build
```
Re-running is idempotent (plain `/fill` + `/setblock`).

### Refreshing after adding blocks
```bash
# after a build/datagen has run:
cd minekea/common/src/main/generated/assets/minekea/blockstates
find . -name '*.json' | sed 's|^\./||;s|\.json$||' | sort \
  > /path/to/minekea/demo-world/block_inventory.txt
# then (if jar contents changed) python extract_jar_contents.py, then:
python generate_layout.py
```
A brand-new material just needs a rule in `canonical()` (and a spot in `ORDER`); an
unmapped material raises an error rather than silently disappearing.
