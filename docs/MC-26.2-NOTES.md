# MC 26.2 port notes

Gotchas and non-obvious API changes hit while porting the mods to Minecraft 26.2 (from 26.1.2, and from
the earlier Yarn/1.21.x line). Not exhaustive â€” see `UPDATE-PLAN.md` for the migration runbook.

## Datagen: "Components not bound yet"

Item data components are bound **lazily during a `ReloadableServerResources` reload**, not at
bootstrap. The `Item` constructor now just registers an initializer into
`BuiltInRegistries.DATA_COMPONENT_INITIALIZERS`; the map is baked and bound later. Datagen never
performs that reload, so any generator reading components at datagen time (e.g.
`Item.getDefaultMaxStackSize()`) throws `NullPointerException: Components not bound yet`
(`Holder$Reference.components`). **This is not a logic bug in the generator** â€” pre-26.2 code that
queried components during datagen silently broke on the update.

Fix â€” bind once, up front, the same way the server reload does. Put it at the **top of
`buildRecipes()`** (`minekea/fabric/.../data/ModDataGenerator.java`):

```java
BuiltInRegistries.DATA_COMPONENT_INITIALIZERS.build(registryLookup)
    .forEach(pending -> pending.apply());
```

`registryLookup` is the datagen `HolderLookup.Provider`. All APIs are public (no access widener).
Vanilla's `RegistryComponentsReport` and `ReloadableServerResources` use the same
`build(provider).forEach(PendingComponents::apply)` pattern. If another datagen provider (item models,
tags, loot) reads components before recipes run, hoist this bind earlier / into that provider. The unit-
test equivalent (baking components in JUnit bootstrap) is in `docs/TESTING.md`.

## Notable API renames (26.1.2 â†’ 26.2)

Registry-constant holders moved into pluralized/util classes (mirroring the Blocks/BlockIds ID split).
The *type* classes keep their names; the **static-instance holders** moved:

- `BlockEntityType.X` â†’ `BlockEntityTypes.X` (`net.minecraft.world.level.block.entity.BlockEntityTypes`). `BlockEntityType<T>` stays.
- `EntityType.X` (e.g. `VILLAGER`, `ITEM`) â†’ `EntityTypes.X` (`net.minecraft.world.entity.EntityTypes`). `EntityType<T>` stays.
- `net.minecraft.advancements.CriteriaTriggers` â†’ `net.minecraft.advancements.triggers.CriteriaTriggers`
  (predicates moved too: `advancements.criterion.*` â†’ `advancements.predicates.*`).
- `LevelRenderer.getLightCoords(...)` / `LevelRenderer.BrightnessGetter` â†’
  `net.minecraft.util.LightCoordsUtil.getLightCoords(...)` / `LightCoordsUtil.BrightnessGetter`.
- `Minecraft.renderNames()` removed â†’ HUD hidden = `Minecraft.getInstance().gui.hud.isHidden()`.
- `ValueInput.contains(String)` removed â†’ `view.getString("k").isPresent()` (read with `getStringOr`/`getBooleanOr`).

### Other removals/changes

- `net.minecraft.util.Tuple` **removed** â†’ `com.mojang.datafixers.util.Pair` (`getA()`â†’`getFirst()`,
  `getB()`â†’`getSecond()`, `new Tuple<>(a,b)`â†’`Pair.of(a,b)`).
- Architectury 21.x `RegistrySupplier.getRegisteredName()` **removed** â†’ `getId().toString()`. (Vanilla
  `Holder.getRegisteredName()` still exists â€” leave those.)
- `ItemStack(RegistrySupplier<Item>)` no longer resolves â†’ call `.get()`.
- `EntityType.byString` removed â†’ `Optional.ofNullable(Identifier.tryParse(id)).flatMap(BuiltInRegistries.ENTITY_TYPE::getOptional)`.
- `BlockPos.getCenter()` removed â†’ `blockPos.distToCenterSqr(vec3)` for squared distance.
- `MenuScreens.register` + `MenuScreens$ScreenConstructor` are private â†’ reach via access widener
  (Architectury 21.x `MenuRegistry` has no screen-factory helper). `TextureSlot.create(String)` is also
  private â†’ access widener.

### Datagen (Fabric) signature changes

- `TagAppender<T, T>` (two type args) â†’ `TagAppender<T>` (single). Affects every
  `configureBlockTags`/`configureItemTags` signature.
- `TagAppender<T>.add(...)` now takes `ResourceKey<T>`, not the `Block`/`Item` object:
  `.add(BLOCK)` â†’ `.add(BLOCK.builtInRegistryHolder().key())`;
  `.add(block.asItem())` â†’ `.add(block.asItem().builtInRegistryHolder().key())`.
- `FabricTagProvider.valueLookupBuilder(TagKey<T>)` renamed to `builder(TagKey<T>)`.
- Predicate classes moved: `net.minecraft.advancements.criterion.{ItemPredicate,StatePropertiesPredicate}`
  â†’ `net.minecraft.advancements.predicates.*`.

## Reading decompiled vanilla source

The 26.2 `-sources.jar` in the Loom cache is an **empty placeholder** (`genSources` was never run for
26.2). To read vanilla implementation code:

1. Use the binary jar (Mojang-mapped, merged, with Fabric `@Environment` annotations):
   `~/.gradle/caches/fabric-loom/minecraftMaven/net/minecraft/minecraft-merged-deobf/26.2/minecraft-merged-deobf-26.2.jar`
2. For signatures only, `javap -p` on the extracted `.class` is fastest.
3. For full bodies, decompile with Vineflower: download it from Maven Central (in Git Bash `curl` fails
   with exit 35 â€” use PowerShell `Invoke-WebRequest`), unzip the classes you need, then
   `java -jar vineflower.jar <classes> <outdir>`.

## The "Client shutdown from post-main" watchdog crash (spurious)

Since the 26.2 update, crash reports titled `Description: Client shutdown from post-main` /
`java.lang.Error: Watchdog (Client shutdown from post-main)` appear on client exit across several mods.
**These are not gameplay crashes and not bugs in this repo's code.** They fire from Mojang's
`ClientShutdownWatchdog` *after* `main()` returns: two non-daemon fixed thread pools
(`pool-1-thread-*`, `pool-2-thread-*`, sized to `availableProcessors()`) park idle in
`LinkedBlockingQueue.take()` and are never `shutdown()`, so the JVM can't exit and the watchdog
force-kills it. archaeology-tweaks and chimeric-lib were verified to create zero runtime threads â€” the
leak is in a shared dependency (architectury / fabric-api / YACL / modmenu / fabric-language-kotlin).
The game does fully close; impact is a spurious report + non-zero exit. Don't chase it in mod code.
